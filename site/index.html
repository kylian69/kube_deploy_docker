Contexte

Mission

Vous êtes une équipe Platform Engineering mandatée par la scale-up fictive NovaPay pour construire leur infrastructure de production from scratch. NovaPay traite des paiements en ligne : disponibilité, sécurité de la supply chain et auditabilité sont des contraintes réelles.

L'objectif est de produire une plateforme complète, documentée et opérable, comme si elle allait réellement accueillir du trafic. Chaque choix doit être justifié par un ADR ; chaque composant de sécurité doit être prouvé fonctionnel par un test ou une démo live.

⚡ Ce qui a changé depuis 2025 — pourquoi cet énoncé est différent
  • Kubernetes 1.35/1.36 (Haru) est la version stable — les features GA de 2024-2025 (sidecar containers, in-place resize, ValidatingAdmissionPolicy) sont maintenant attendues
  • Istio Ambient Mesh est GA depuis Istio 1.22 — le mode sidecar n'est plus le choix par défaut en 2026, le projet impose d'évaluer et justifier le choix
  • Cilium + Gateway API remplace Nginx Ingress : la Gateway API Kubernetes est le standard réseau désormais, Ingress est legacy
  • Grafana Alloy (successeur de Grafana Agent) remplace Promtail + Agent — pipeline OTEL unifié métriques/logs/traces/profiling en un seul collector
  • OpenTelemetry est diplômé CNCF et à 95 %+ d'adoption — l'instrumentation OTel est imposée, pas optionnelle
  • SLSA v1.1 est la référence supply chain — Sigstore/Cosign + attestations de provenance font partie du minimum attendu en industrie réglementée
  • Kubescape 4.0 remplace kube-bench comme outil de scanning de posture (runtime + AI agent scanning inclus)
  • Backstage IDP : le Platform Engineering implique maintenant un Internal Developer Portal — au moins un template de service catalogué
  • EPSS scoring : prioriser les CVE par probabilité d'exploitation réelle, pas juste par score CVSS
Infrastructure disponible

Le matériel & le découpage cible

L'infrastructure n'est pas une abstraction : vous opérez sur du matériel réel, avec ses contraintes réelles. Le réalisme du projet vient de là.

🖥️
2× Serveur compute
40 cores / 192 Go RAM chacun. Hôtes des workers et de la majorité de la charge. Virtualisés via Proxmox.
💻
2–3× PC légers
4 cores / 8 Go RAM. Rôle d'arbitre etcd (control-plane léger) et de générateurs de charge pour le chaos.
☁️
Budget cloud ciblé
DNS managé, object storage S3/GCS, et un cluster managé éphémère pour la démo de portabilité.
⚖️ La contrainte qui dicte l'architecture : le quorum etcd

etcd exige un nombre impair de membres (idéalement 3) pour maintenir le quorum. Avec seulement 2 gros serveurs, un control-plane sur chacun donnerait un cluster moins résilient qu'avec un seul (perte d'un membre = perte du quorum). La solution réaliste : un PC léger sert de 3ᵉ control-plane / arbitre etcd. Un control-plane consomme peu — 4 cores / 8 Go suffisent. C'est un pattern de prod courant : nœuds de contrôle légers, workers costauds.

── DÉCOUPAGE PHYSIQUE (Proxmox sur les 2 gros serveurs) ── Serveur A (40c / 192Go) Serveur B (40c / 192Go) PC léger 1 (4c/8Go) ├─ VM cp-prod-1 (4c/8) ├─ VM cp-prod-2 (4c/8) └─ cp-prod-3 (arbitre etcd) ├─ VM worker-prod-1 (14c/56) ├─ VM worker-prod-2 (14c/56) ├─ VM worker-prod-3 (14c/56) ├─ VM cp-staging (4c/8) PC léger 2/3 (4c/8Go) └─ VM worker-staging (6c/24) └─ VM worker-staging (6c/24) └─ générateurs de charge (k6 / Locust) CLUSTER PROD : 3 control-plane (HA réel, répartis sur 3 machines physiques) + 3 workers costauds CLUSTER STAGING : 1 control-plane (non-HA assumé — décision d'archi documentée) + 2 workers CHARGE / CHAOS : PC légers en générateurs de trafic, hors cluster
🎯 Décision d'architecture : prod en HA, staging volontairement léger

Le cluster prod est en HA strict. Le staging est en single control-plane — un choix assumé, pas une limite subie. De nombreuses entreprises ont un staging non-HA pour économiser. Savoir justifier et documenter ce trade-off dans un ADR est en soi une compétence d'architecte attendue à ce niveau.

Usage ciblé du cloud — le réalisme hybride sans gaspillage

Le cloud n'est pas là pour dupliquer ce que le matériel fait déjà gratuitement, mais pour ce qui est intrinsèquement externe en entreprise :

DNS + TLS
Domaine réel + DNS managé (Route53/Cloud DNS) → vrais certificats Let's Encrypt en DNS-01. Quelques €/mois.
Object Storage
Bucket S3/GCS comme backend Velero + Loki + Tempo. On ne stocke jamais ses backups sur le cluster sauvegardé.
Cluster managé éphémère
Petit EKS/GKE allumé quelques jours pour la démo de portabilité GitOps, puis détruit. Facturé à l'usage.
FinOps
Suivi du coût réel et arbitrage on-prem (capacité fixe payée) vs cloud (élastique facturé), documenté.

Phases du projet

Travail demandé

PHASE 01 — Semaines 1–2
Infrastructure as Code & Bootstrap Multi-Cluster
Tout doit être reproductible : Terraform + Ansible doivent provisionner les VMs Proxmox et bootstrapper les deux clusters depuis zéro.
  • Installer Proxmox sur les 2 gros serveurs, provisionner les VMs via le provider Terraform Proxmox infra
  • Bootstrap du cluster prod en HA : 3 control-plane sur 3 machines physiques distinctes (dont 1 PC léger comme arbitre etcd) infra
  • Bootstrap du cluster staging : single control-plane, choix non-HA assumé et documenté en ADR infra
  • Kubernetes 1.35 via kubeadm, configuration HA avec endpoint API load-balancé (kube-vip ou HAProxy)
  • Installer Cilium comme CNI sur les deux clusters avec Gateway API activée — pas d'Ingress Controller
  • Configurer Cilium LB-IPAM ou MetalLB pour les LoadBalancer Services on-prem
  • Déployer ArgoCD sur le cluster prod, l'enregistrer comme hub multi-cluster gérant prod ET staging infra
  • Pattern App of Apps + ApplicationSets avec générateur de cluster (un même chart, deux destinations)
  • Mettre en place le DNS managé cloud + cert-manager DNS-01 pour de vrais certificats TLS cloud
  • Scan Checkov de tout le code IaC en CI
  • Bootstrap runbook : recréer l'intégralité de la plateforme en moins d'une heure
PHASE 02 — Semaines 2–3
Supply Chain Security & Hardening — SLSA v1.1
Sécuriser la chaîne de build avant tout déploiement. En 2026, les attestations de provenance sont attendues en industrie réglementée.
  • Pipeline CI avec Trivy — prioriser par EPSS score, pas seulement CVSS 2026
  • Bloquer tout CVE CRITICAL avec EPSS > 0.1 non justifiée
  • Signature d'image Cosign + génération d'attestations de provenance SLSA Build Level 2 UPD
  • SBOM CycloneDX généré par Syft à chaque build et archivé dans le registry OCI
  • Vérification de signature à l'admission via Kyverno policy (blocage si signature absente)
  • Kubescape 4.0 : scan posture CIS Benchmark + NSA/CISA, corriger tous les findings HIGH 2026
  • ValidatingAdmissionPolicy Kubernetes native pour les règles simples (feature GA en 1.30+) 2026
  • External Secrets Operator + Vault : aucun secret dans Git, zéro SealedSecret statique
  • NetworkPolicy Cilium L7 : deny-all par défaut, autoriser uniquement les flux DNS/HTTP documentés
PHASE 03 — Semaines 3–4
Application NovaPay — Déploiement & Service Mesh Ambient
Déployer les micro-services avec Istio Ambient Mesh. Justifier le choix ambient vs sidecar dans un ADR.
  • Packager chaque service en Helm chart avec values staging/prod (Kustomize pour les overlays env)
  • Activer Istio Ambient sur les namespaces prod : ztunnel L4 + waypoint proxy L7 sur payment-svc 2026
  • Rédiger l'ADR ambient vs sidecar : performance, ressources, observabilité, trade-offs 2026
  • Configurer HTTPRoutes et GRPCRoutes via Gateway API (plus d'annotations Ingress) 2026
  • Canary deployment via Argo Rollouts + Gateway API HTTPRoute (weight-based, 10% → 50% → 100%)
  • Sidecar containers natifs Kubernetes 1.28+ pour init patterns (logger, config-reloader) 2026
  • HPA + KEDA sur les services critiques (métriques custom RPS via OTEL)
  • PodDisruptionBudgets, topologySpreadConstraints, liveness/readiness/startup probes obligatoires
  • TLS automatique via cert-manager + Let's Encrypt ou CA interne
PHASE 04 — Semaines 4–5
Observabilité Unifiée — OTEL + Grafana Alloy + SLO
Pipeline d'observabilité unifié OpenTelemetry. Grafana Alloy remplace Promtail + Grafana Agent.
  • Déployer Grafana Alloy en DaemonSet : collecte métriques (Prometheus scrape), logs et traces en un seul agent 2026
  • Instrumenter les services NovaPay avec le SDK OpenTelemetry (OTLP vers Alloy → Mimir/Loki/Tempo)
  • Configurer Istio Ambient pour exporter les métriques L7 via waypoint proxies en OTLP 2026
  • Profiling continu avec Pyroscope ou Grafana Profiling (Alloy intégration native) 2026
  • Dashboard Grafana : SLI/SLO payment-svc (latency p99, error rate, saturation)
  • 8+ alertes Alertmanager pertinentes avec runbook lié dans chaque annotation
  • SLO défini et mesuré : 99.9% dispo, p99 latency < 300ms (budget d'erreur calculé)
  • Dashboard on-call one-pager : 90% des incidents diagnosticables en 1 écran
  • Corréler log → trace → métrique via Grafana Explore (exemplars)
PHASE 05 — Semaine 5–6
Sécurité Runtime, eBPF & Threat Modelling
Détection et réponse en temps réel. Cilium eBPF comme source de vérité réseau.
  • Falco avec règles custom (shell dans pod payment, accès /proc/*/mem, exfiltration DNS)
  • Cilium Tetragon pour la détection eBPF bas niveau (syscall audit, file access) 2026
  • Router alertes Falco + Tetragon → Alertmanager → Slack avec enrichissement context pod
  • Audit Logs Kubernetes exportés vers Loki, requêtes LogQL pour détecter privilege escalation
  • Kubescape 4.0 runtime : scan posture continu, pas juste au deploy 2026
  • Simuler 3 scénarios d'attaque et démontrer la détection : container escape simulé, secret theft via env, lateral movement
  • Threat modelling STRIDE de l'architecture NovaPay (documenter surface d'attaque résiduelle)
  • RBAC minimal : least privilege sur tous les ServiceAccounts, pas de ClusterAdmin inutile
PHASE 06 — Semaine 6–7
Chaos Engineering & Résilience
Prouver que la plateforme tient sous la pression et respecte les SLO pendant les pannes.
  • 5 GameDays documentés avant exécution (hypothèse, SLO cible, critères de succès)
  • Générer une charge réaliste depuis les PC légers (k6 ou Locust) sur payment-svc pendant les tests infra
  • Chaos Mesh ou LitmusChaos : pod kill, network delay/partition, node drain, disk fill
  • Tuer une VM control-plane prod et prouver que le quorum etcd tient (HA validé) infra
  • Valider que les SLO restent respectés pendant chaque test (budget d'erreur non dépassé)
  • Tester le canary Argo Rollouts : déclencher un rollback automatique si error rate > 5%
  • Velero vers le bucket cloud S3/GCS : backup namespace, suppression, restauration complète prouvée cloud
  • Tester la résilience Vault (5 min d'indisponibilité) : que deviennent les pods qui démarrent ?
  • Chaos Report : RCA structuré pour chaque test, corrective actions documentées
PHASE 07 — Semaine 7
Platform Engineering — Internal Developer Portal
En 2026, 80% des grandes orgs ont une équipe Platform Engineering. L'IDP est le produit qu'elle livre aux devs.
  • Déployer Backstage (IDP) dans le cluster via Helm 2026
  • Cataloguer les 3 services NovaPay : owner, tier, SLO, runbook, repo, dashboard Grafana
  • Créer un Software Template Backstage : "Je veux un nouveau microservice" → génère repo, pipeline CI, chart Helm, namespace k8s, entry Backstage 2026
  • TechDocs : documentation des services rendue dans Backstage depuis les repos Git
  • Score de maturité par composant (SLO défini ? observabilité ? runbook ? scan sécurité ?)
  • DORA Metrics visibles dans Backstage : deployment frequency, lead time, MTTR, change failure rate 2026
PHASE 08 — Semaine 7–8
FinOps & Portabilité Hybride Cloud
Démontrer que la plateforme est portable et que vous savez arbitrer le coût — le raisonnement clé d'un platform engineer en 2026.
  • Provisionner un petit cluster managé (EKS/GKE) via Terraform, allumé quelques jours seulement cloud
  • Déployer la même app via le même ArgoCD pour prouver la portabilité GitOps on-prem ↔ cloud cloud
  • Documenter ce qui a dû changer (StorageClass, LoadBalancer, IAM) et ce qui est resté identique
  • Mettre en place un suivi de coût : OpenCost ou Kubecost sur le cluster pour la visibilité par namespace FinOps
  • Rapport FinOps : coût réel cloud du mois, arbitrage on-prem (capacité fixe payée) vs cloud (élastique) FinOps
  • Rightsizing : Goldilocks / VPA en mode recommendation, identifier les requests/limits sur/sous-dimensionnés
  • Détruire le cluster managé après la démo et prouver le coût maîtrisé (facturation à l'usage)
PHASE 09 — Semaine 8
Documentation, ADRs & Soutenance
Une plateforme non documentée n'existe pas. Le jury s'attend à pouvoir opérer le système sans vous.
  • 7+ Architecture Decision Records : staging non-HA assumé, arbitre etcd sur PC léger, Ambient vs sidecar, Cilium CNI, Alloy vs OTel Collector, SLSA Level, arbitrage on-prem vs cloud infra
  • Runbooks opérationnels : control-plane KO, perte d'un serveur physique, Vault indisponible, node NotReady, canary bloqué
  • Rapport de sécurité : modèle de menace STRIDE, findings Kubescape résolus, surface résiduelle
  • Présentation 30 min : architecture → live demo déploiement GitOps d'un hotfix → chaos demo (tuer une VM CP)
  • Chaque membre du groupe doit pouvoir défendre n'importe quel composant face au jury

Règles non négociables

Contraintes absolues

🚫
Zéro secret dans Git
Tout secret détecté dans un commit (y compris dans l'historique) = pénalité immédiate. ESO + Vault obligatoires.
🚫
Zéro kubectl apply manuel en prod
ArgoCD est le seul acteur autorisé à modifier l'état du cluster de prod. Toute déviation visible dans ArgoCD doit être auto-corrigée (sync policy).
🚫
Zéro image sans digest
Toutes les images référencées par leur SHA256 digest (pas de tag mutable, pas de :latest). Kyverno bloque toute image sans digest.
🚫
Zéro Ingress Controller
Gateway API Kubernetes uniquement. Nginx/Traefik Ingress = pratique legacy en 2026. Justifiez tout écart dans un ADR.
✅
Tout doit être démontrable en live
Le jury peut demander une démo de n'importe quelle partie à la soutenance : déploiement GitOps, rollback canary, détection Falco, restauration Velero.
